⚔ Phase 1 & 2 — The Attack
AES-CBC with PKCS#7 padding leaks information through HTTP response codes. The attacker needs only the IV and ciphertext to recover the full plaintext — no key required. Average ~128 queries per byte.
🛡 Phase 3 — The Defense
AES-GCM uses authenticated encryption. The 128-bit auth tag is verified before decryption begins. Every tampered request returns the same HTTP 403 — no oracle signal, attack impossible.

Executive Summary

This project demonstrates the Padding Oracle Attack against AES-CBC encryption and its complete defense using AES-GCM. The attack exploits the combination of CBC mode and leaky padding error responses to recover plaintext without the key. Phase 3 proves that authenticated encryption (AES-GCM) eliminates the attack entirely.

Phase 1 — Vulnerable Server

AES-128-CBC server on port 5000. Returns HTTP 200 for valid padding, HTTP 403 for invalid — creating a padding oracle that leaks one byte of information per up to 256 HTTP requests.

Phase 3 — The Defense

Conclusions & Recommendations

1. Never use AES-CBC for new systems — use AES-GCM or another AEAD cipher.
2. Always authenticate before decrypting — verify integrity first.
3. Return generic errors — never differentiate failure modes in responses.
4. Use constant-time comparisons to prevent timing oracles.

Real-world impact: POODLE (CVE-2014-3566), Lucky Thirteen (CVE-2013-0169), ASP.NET Padding Oracle (CVE-2010-3332).
