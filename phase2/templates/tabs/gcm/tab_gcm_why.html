This tab explains exactly why the padding oracle attack that works perfectly against AES-CBC produces zero results against AES-GCM.
💡
🧠 The One-Sentence Answer
AES-GCM has no padding — and every tampered ciphertext is rejected by the authentication tag before decryption even starts, so the server never produces a different response for different inputs. The oracle is silent.

🔄 What Happens Step by Step

Against CBC — The Attack Works
Attacker crafts ciphertext with guess byte at position [15] ↓ CBC server decrypts using secret key ← decryption happens first ↓ Server checks PKCS#7 padding ↓ Padding valid → HTTP 200 ← oracle signal! Padding invalid → HTTP 403 Attacker reads the difference → byte recovered → repeat × 16 → full plaintext

2
Against GCM — The Attack Fails
Attacker crafts ciphertext with tampered bytes ↓ GCM server recomputes GHASH tag over received ciphertext ↓ Tag mismatch → HTTP 403 immediately ← decryption never runs ↓ All 256 guesses → all HTTP 403 Attacker sees identical responses → nothing learned → attack fails

Why CBC Is Vulnerable
CBC was designed only to encrypt — not to authenticate. It has no way to detect tampering. When an attacker sends crafted ciphertext, the server decrypts it faithfully and reveals padding information through its response. That information leak is the oracle.

The vulnerability is not in AES itself — it is in the combination of CBC mode + padding checks + different error responses.
Why GCM Is Safe
GCM uses authenticated encryption — it produces a 128-bit tag that is a cryptographic proof that the ciphertext is unmodified. Any tampered input produces a tag mismatch, and the server rejects it with the same generic error every time.

There is no padding to exploit, and no difference in responses to observe. The oracle cannot exist.
