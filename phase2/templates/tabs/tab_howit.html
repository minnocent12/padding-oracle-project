🧠 What Is a Padding Oracle Attack?
A Padding Oracle Attack is a chosen-ciphertext attack that lets an attacker decrypt any AES-CBC ciphertext without knowing the secret key.

The word "oracle" means an information source — in this case, the vulnerable server itself. Every time the server says "valid padding" or "invalid padding" in response to a crafted request, it leaks one bit of information about the plaintext. Repeat this up to 256 times per byte, and the full plaintext is revealed.

Important: AES itself is not broken. The vulnerability is in CBC mode + leaky error handling.
📋
🎯 What the Attack Needs to Start
Three things are required — nothing more:

1 A valid IV + Ciphertext — intercepted from network traffic, stolen from a cookie, extracted from a URL parameter, or obtained from a database leak.

2 Access to the vulnerable server — a server that decrypts attacker-supplied ciphertext and returns different responses for valid vs invalid padding (HTTP 200 vs 403, different error messages, or even timing differences).

3 Knowledge of the block size — AES always uses 16-byte blocks (128-bit). This is publicly known and never secret.

🔄 The Attack — Step by Step

1
Encryption (Setup)
The server encrypts a plaintext message using AES-CBC with a secret key and a random IV. The attacker intercepts or obtains the IV and Ciphertext. The key is never seen.
Plaintext → [AES-CBC Encrypt with secret key] → IV + Ciphertext Attacker obtains: IV = f8800b32... CT = 25fc7171...
▼
2
Crafting a Fake Block
The attacker creates a crafted 16-byte block and injects a guess value at the target byte position. All other bytes are set to produce the desired PKCS#7 padding value once decrypted.
For byte [15] (last byte), target padding = 0x01: crafted[15] = guess ← try 0x00, 0x01, ..., 0xFF crafted[0..14] = 0x00 ← don't care yet Send to server: POST /decrypt { iv: crafted, ciphertext: real_ct_block }
▼
3
Reading the Oracle
The server decrypts the block using the real key and checks the padding. It leaks the result through its HTTP response code:
HTTP 403 → Padding invalid → Wrong guess, try next value HTTP 200 → Padding valid! → This guess produced 0x01 padding ✔ Repeat up to 256 times per byte (average ~128 guesses)
The attacker sends up to 256 HTTP requests per byte until the server responds HTTP 200.
▼
4
XOR Math — Recovering the Byte
Once a valid guess is found, two XOR operations reveal the plaintext byte:
Step 1: intermediate = guess ⊕ pad_value 0xa1 ⊕ 0x10 = 0xb1 (intermediate byte) Step 2: plaintext = intermediate ⊕ prev_block[byte_idx] 0xb1 ⊕ 0xf8 = 0x49 → 'I' ✔ Why this works: AES-CBC decryption XORs with the previous block internally. By controlling the crafted block we reverse that XOR to find the plaintext.
▼
5
Full Recovery
Repeat for all 16 bytes per block (right to left), then for every block. Strip PKCS#7 padding from the end. The full plaintext is recovered.
Attack order: byte[15] → byte[14] → ... → byte[0] → next block... Result: "I would like to" ← recovered without key Cost: ~128 queries × 16 bytes = ~2048 HTTP requests Key: Never used. Never needed.

⚠️ Why Is This Dangerous?

Real-World Data at Risk
Any data a server encrypts with AES-CBC and decrypts on request is vulnerable — session cookies containing usernames and roles, authentication tokens, encrypted API payloads, medical records, and financial data.
Where Attackers Get IV + Ciphertext
Browser cookies — visible in DevTools → Application → Cookies.
Network traffic — intercepted with Burp Suite or Wireshark.
URL parameters — some apps embed encrypted tokens in URLs.
Database leaks — encrypted columns come with IVs attached.
API responses — tokens returned openly in JSON responses.

🛡️ How to Fix It

Use Authenticated Encryption
Replace AES-CBC with AES-GCM (Galois/Counter Mode). GCM includes a built-in authentication tag — any tampering with the ciphertext is detected before decryption even begins, eliminating the oracle entirely.
Other Mitigations
Encrypt-then-MAC — apply an HMAC over the ciphertext and verify it before decryption. If the MAC fails, always return the same generic error.

Constant-time responses — never reveal whether padding or authentication failed through different error messages or timing. Always return the same opaque error regardless of failure reason.

📜 Real-World Attacks Using This Technique

POODLE (2014)
Exploited SSL 3.0's CBC padding to decrypt HTTPS cookies. Forced the entire internet to deprecate SSL 3.0. CVE-2014-3566
Lucky Thirteen (2013)
A timing-based padding oracle against TLS 1.0/1.1/1.2 using CBC mode. Even "constant-time" implementations were vulnerable. CVE-2013-0169
ASP.NET Padding Oracle (2010)
Allowed attackers to decrypt ViewState and authentication cookies in ASP.NET web applications, exposing session data of millions of sites. CVE-2010-3332
