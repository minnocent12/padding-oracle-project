🧠 What Is AES-GCM?
AES-GCM (Galois/Counter Mode) is an authenticated encryption algorithm. Unlike AES-CBC which only encrypts, GCM both encrypts and authenticates — it produces a cryptographic proof that the ciphertext has not been tampered with.

This single property is what makes padding oracle attacks impossible against GCM. There is no padding, and any tampered ciphertext is rejected before decryption begins.

🔄 How AES-GCM Works — Step by Step

1
Random Nonce Generation
Every encryption starts with a fresh 96-bit (12-byte) random nonce. The nonce is never reused — reusing a nonce with the same key would catastrophically break GCM's security guarantees.
nonce = os.urandom(12) # 96-bit, unique per message Unlike CBC's IV, the GCM nonce does not need to be unpredictable — it only needs to be unique. But random is the safest choice.
▼
2
AES-CTR Encryption (No Padding)
GCM uses AES in Counter (CTR) mode — a stream cipher mode that generates a keystream and XORs it with the plaintext. No padding is required because CTR mode works on arbitrary-length data.
Plaintext: "Hello" (5 bytes) CTR generates a keystream: K1 K2 K3 K4 K5 Ciphertext = Plaintext ⊕ Keystream No PKCS#7 padding. No padding oracle. No vulnerability.
▼
3
GHASH — Computing the Auth Tag
Simultaneously with encryption, GCM computes a 128-bit authentication tag using the GHASH function over the ciphertext and the key.
GHASH(key, ciphertext) → 128-bit tag The tag is a mathematical fingerprint of: - The ciphertext - The secret key - The nonce Change any one of these → completely different tag Tag matches → ciphertext is authentic and unmodified
▼
4
Decryption — Authenticate BEFORE Decrypt
On decryption the server recomputes the tag first and compares it to the received tag in constant time. Decryption only happens if they match.
Received: nonce + ciphertext + tag Step 1: recompute_tag = GHASH(key, received_ciphertext) Step 2: compare recompute_tag vs received_tag (constant time) Mismatch → HTTP 403 generic error — decryption never runs Match → AES-CTR decrypt → return plaintext

🛡️ Why Padding Oracle Is Impossible Against GCM

AES-CBC — How the Oracle Works
1. Attacker sends crafted ciphertext
2. Server decrypts with secret key
3. Server checks PKCS#7 padding
4. Server returns HTTP 200 (valid padding) or 403 (invalid padding)
5. Attacker reads response → one bit of info leaked
6. Repeat 256× per byte → full plaintext recovered

The oracle exists because the server decrypts first, then checks, then reveals the result.
AES-GCM — Why There Is No Oracle
1. Attacker sends crafted ciphertext
2. Server recomputes GHASH tag
3. Tag mismatch → HTTP 403 immediately
4. Server never decrypts tampered data
5. Attacker reads response → same 403 every time
6. 256 guesses × 256 bytes → all 403 → nothing learned

No padding means no padding oracle. The auth tag catches every tamper attempt before decryption.

🎓 Key Security Lessons

Lesson 1 — Always Authenticate
Encryption without authentication is incomplete security. AES-CBC encrypts but does not authenticate — that gap is exactly what the padding oracle exploits. AES-GCM does both in one pass. Use AES-GCM
Lesson 2 — Authenticate Before Decrypt
Always verify integrity before processing data. GCM checks the tag first — the server never touches tampered ciphertext. This principle eliminates entire classes of attacks beyond just padding oracles. Auth-then-decrypt
Lesson 3 — Same Error Always
The GCM server returns the identical 403 for every failure — wrong tag, wrong nonce, missing fields. Never differentiate errors in ways the attacker can observe. Constant-time comparison prevents timing oracles too. Generic errors
