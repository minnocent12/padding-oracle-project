🛡️ Why the Attack Failed
Every single one of those 256 requests returned HTTP 403 — the exact same response, with the exact same message.

In Phase 2, the CBC server returned HTTP 200 for the one guess that produced valid padding, giving the attacker a signal to exploit. Here, GCM returns 403 for all 256 guesses — no signal, no information, no oracle.

The auth tag verification happens before decryption. The server never even attempts to decrypt tampered ciphertext. There is nothing for the attacker to learn.

🔑 The Core Difference: Authenticated Encryption
AES-CBC is encryption only — it scrambles data but provides no guarantee that the ciphertext hasn't been modified.

AES-GCM is authenticated encryption — it both encrypts the data and produces a 128-bit authentication tag that acts as a cryptographic proof of integrity. If even one bit of the ciphertext, nonce, or associated data changes, the tag verification fails and decryption is never attempted.
▼

🔄 How AES-GCM Works

1
Encryption
AES-GCM uses a random 96-bit nonce (never reused) and encrypts in CTR mode. Simultaneously it computes a GHASH over the ciphertext to produce the auth tag.
Plaintext → [AES-CTR encrypt] → Ciphertext ↓ [GHASH over ciphertext] ↓ Output: Nonce + Ciphertext + Auth Tag (128-bit)
▼
2
Decryption — Tag Verified FIRST
Before any decryption happens, the server recomputes the GHASH over the received ciphertext and compares it to the received tag. Only if they match does decryption proceed.
Received: Nonce + Ciphertext + Tag Step 1: Recompute GHASH over received ciphertext Step 2: Compare computed tag vs received tag Mismatch? → Reject immediately, return generic 403 Match? → Decrypt and return plaintext Key point: decryption never runs on tampered data
▼
3
Why the Padding Oracle Cannot Work
The padding oracle attack works by crafting modified ciphertext and observing whether the server responds differently for valid vs invalid padding. In GCM this is impossible because:
Attacker crafts tampered ciphertext ↓ Server recomputes GHASH → tag mismatch ↓ Server returns HTTP 403 — generic error, no padding info ↓ Attacker sends next guess → same result: HTTP 403 ↓ All 256 guesses → all HTTP 403 → no signal → no oracle There is no padding in GCM. There is no padding error to leak. The tag check eliminates all information before it can escape.

🏗️ CBC vs GCM Architecture

AES-CBC (Vulnerable)
Mode: Cipher Block Chaining
Padding: PKCS#7 (required, exploitable)
Auth: None — no integrity check
Error: Different for padding valid vs invalid
Oracle: Yes — server leaks padding info

CBC encrypts blocks sequentially, XORing each block with the previous ciphertext block. This XOR dependency is exactly what the attacker exploits to recover plaintext byte by byte.
AES-GCM (Secure)
Mode: Galois/Counter Mode
Padding: None — stream cipher mode
Auth: 128-bit GHASH tag (mandatory)
Error: Always the same generic 403
Oracle: Impossible — tag blocks all probing

GCM uses CTR mode (no padding needed) and a GHASH function that binds the ciphertext to the key. Any modification invalidates the tag. The server never decrypts tampered data.

🎓 Key Lessons from Phase 3

Lesson 1 — Encrypt-then-Authenticate
Never use encryption without authentication. AES-CBC encrypts but does not authenticate — this is the root cause of the vulnerability. AES-GCM does both in a single pass. Use AES-GCM
Lesson 2 — Generic Errors Always
Even if you use CBC, returning the same error for every failure prevents the oracle. The GCM server always returns the identical 403 response — no timing difference, no message difference, no oracle. Constant-time
Lesson 3 — Never Decrypt Untrusted Data
The GCM server verifies the auth tag before decryption. This principle — authenticate before processing — is fundamental to secure system design and eliminates entire classes of attacks. Auth first
